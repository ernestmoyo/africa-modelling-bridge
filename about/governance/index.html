The argument

The platform is modular.

Every feature ships as a discrete module with its own owner, budget, surface, and on/off switch. A funder can sponsor a single module. If any module is turned off, the rest still build. This is the funder protection — and it is the operational protection too, because it means a delay on one piece never blocks the others.

The twenty modules fall into four families: Data spine (the public information layer), People (researchers and funders made routable), AI-powered (frontier models applied in narrow, edited loops), and Operations (the governance, distribution and audit layer that makes the rest credible).

Editorial workflow

Git + PR + Decap CMS.

The audit trail is the governance. Every word on the site is reachable as a public commit with an editor, a timestamp, and a diff.

  1. 01

    Draft on a branch

    Editor opens a branch — either via git locally, or via Decap CMS in the browser from v0.1. Drafts are markdown in src/content/.

  2. 02

    Pull request, public

    Every change is a PR against main. AI-generated drafts are tagged and require a human editor before merge.

  3. 03

    Review · methods · facts

    Lead consultant signs off on every PR. Methods content is additionally reviewed by the named working-modeller adviser for that dossier.

  4. 04

    Auto-deploy on merge

    Merge triggers a Netlify build. The audit trail is the git history — every change is signed, dated, and addressable.

Take-down policy

Five business days. Every request logged.

Every document we index is already in the public domain. We still honour ministry take-down requests — within five business days, with a public log of every request and its outcome. Trust takes years to build and one uncorrected mistake to erode. The policy is the floor. M17 implements it.
